# AI 同声传译助手产品设计文档

## 1. 产品概述

### 1.1 产品名称

暂定名：听译窗

英文名：EchoSub

### 1.2 产品一句话

听译窗是一款桌面端 AI 同声传译助手，能将用户正在观看或收听的外语音频实时翻译成用户选择的目标语言字幕，并在上下文变清楚后自动修正前文字幕，帮助用户持续跟上外语内容节奏。

### 1.3 产品定位

听译窗不是大而全的会议翻译平台，而是一个“小而美”的桌面字幕工具。

它面向高频、轻量、单人使用场景：用户正在看外语技术分享、国际发布会、公开课、网课或会议直播时，只需要打开一个悬浮字幕窗，选择源语言和目标语言，就能更轻松地理解内容。

### 1.4 核心价值

- 降低外语听力门槛，让用户更快理解核心信息。
- 支持选择源语言和目标语言，默认使用“自动识别 -> 简体中文”。
- 用悬浮字幕降低打扰，不改变用户原本的观看习惯。
- 通过自动修正机制提升连续内容的理解质量。
- 用极简交互让用户在真实场景中愿意长期依赖。

## 2. 背景与问题

用户经常需要观看英语或其他外语内容，例如：

- 英文技术演讲
- 国际产品发布会
- 海外公开课
- Zoom、Teams、Meet 等线上会议
- YouTube、Bilibili、Coursera 等平台课程
- 本地视频或录播文件

这些内容的问题并不只是“完全听不懂”，更多时候是：

- 听懂单词，但跟不上语速。
- 某一句没反应过来，后面内容就断了。
- 技术词、专有名词、缩写导致理解卡顿。
- 实时字幕识别错误后无法修正，用户会被错误信息误导。
- 普通字幕工具绑定具体平台，无法覆盖任意桌面音频来源。

因此，本产品的重点不是做最复杂的翻译系统，而是做一个最容易被打开、最容易被使用、最不打扰的实时字幕理解辅助层。第一版默认面向中文用户，但不把产品能力写死为英译中。

## 3. 目标用户

### 3.1 核心用户

技术学习者、开发者、学生、产品经理、研究人员，以及经常观看外语专业内容的人。

### 3.2 典型用户画像

用户 A：开发者

- 经常观看英文技术发布会和框架作者演讲。
- 英语阅读能力可以，但听力跟不上原速演讲。
- 希望在不切换网页、不上传视频的情况下获得目标语言字幕，默认是实时中文字幕。

用户 B：学生

- 需要观看海外公开课和英文课程视频。
- 希望快速理解课程内容，不想频繁暂停查词。
- 需要回看最近几分钟字幕，补上刚错过的信息。

用户 C：职场用户

- 偶尔参加外语分享或跨国线上会议。
- 不需要复杂会议纪要，只需要实时知道对方在说什么。
- 更在意操作简单、稳定、低打扰。

## 4. 产品原则

### 4.1 小而美

第一版只解决一个问题：把单向外语音频流实时翻译成用户选择的目标语言，并以字幕方式呈现。

不做大平台，不做账号体系，不做复杂工作台，不做协作功能。

### 4.2 跟得上比完美更重要

实时字幕允许先给出临时结果，再根据上下文修正。用户最需要的是不断跟上内容节奏，而不是等待一句完全确定的翻译。

### 4.3 低打扰

字幕窗应像系统工具一样轻：可置顶、可拖动、可收起，不遮挡主要内容，不要求用户改变原有观看流程。

### 4.4 错了要会改

自动修正是产品核心差异点。系统不能只输出一次性字幕，而要能根据后续语音和上下文更新刚才的识别或翻译错误。

## 5. MVP 范围

### 5.1 必做功能

#### 5.1.1 一键开始听译

用户打开应用后，可以点击“开始听译”，系统开始采集音频并输出目标语言字幕。

MVP 优先支持麦克风输入。系统音频采集作为后续扩展能力。

#### 5.1.2 源语言与目标语言选择

用户可以在开始听译前选择源语言和目标语言。

第一版保持简单：

- 源语言默认：自动识别
- 目标语言默认：简体中文
- 源语言候选：自动识别、英语、日语、韩语
- 目标语言候选：简体中文、英语、日语、韩语

语言选择不做成复杂设置页，而是在主窗口中用两个紧凑下拉框完成：

```text
源语言：自动识别
目标语言：简体中文
```

这样既保留产品的普适性，又不牺牲“小而美”的使用体验。

#### 5.1.3 悬浮字幕窗

提供一个简洁的置顶字幕窗口，显示实时目标语言翻译，默认显示中文字幕。

字幕窗能力：

- 置顶显示
- 可拖动位置
- 支持开始、暂停
- 显示连接或听译状态
- 支持最近一句自动更新

#### 5.1.4 实时临时字幕

系统在音频输入过程中快速显示临时翻译结果，让用户感觉内容正在被持续理解。

临时字幕可以带有轻微视觉状态，例如较浅颜色或“生成中”状态。最终确认后再切换为普通字幕样式。

#### 5.1.5 自动修正上一句

系统应支持更新最近一到三句字幕。

示例：

初始字幕：

```text
我们会把模型放在边缘。
```

修正后：

```text
我们会把模型部署在边缘设备上，以降低实时推理延迟。
```

修正策略：

- 保留最近几句可修改字幕。
- 新的上下文到来后，重新判断前文是否需要修正。
- 修正时替换原句，而不是新增一条重复字幕。
- 在历史面板中保留修正后的最终版本。

#### 5.1.6 字幕历史面板

用户可以打开历史面板，查看最近 1 到 3 分钟的字幕内容。

历史面板用途：

- 补看刚才错过的信息。
- 展示系统持续翻译能力。
- 用于展示产品体验闭环。

### 5.2 可选加分功能

后续可考虑：

- 双语对照字幕模式
- 字幕字体大小调整
- 清空历史
- 复制最近字幕
- 简单术语高亮
- 本地字幕导出为 `.txt`

### 5.3 暂不做功能

MVP 中明确不做：

- 登录注册
- 云端同步
- 多人协作
- 多说话人识别
- 完整会议纪要
- 语音播报
- 复杂语言高级设置，例如地区变体细分、语言自动路由、按应用保存规则
- 专业术语库管理
- 浏览器插件
- 移动端

## 6. 核心用户流程

### 6.1 首次使用流程

1. 用户打开听译窗桌面应用。
2. 应用展示极简主界面。
3. 用户选择源语言和目标语言，默认是“自动识别 -> 简体中文”。
4. 用户点击“开始听译”。
5. 系统请求麦克风权限。
6. 用户播放外语视频或会议音频。
7. 字幕窗开始显示实时目标语言翻译。
8. 用户可以拖动字幕窗到屏幕底部。
9. 系统根据上下文修正刚才的字幕。
10. 用户点开历史面板，查看最近内容。

### 6.2 日常使用流程

1. 用户打开外语内容。
2. 打开听译窗。
3. 确认源语言和目标语言。
4. 点击开始。
5. 看目标语言字幕跟随内容流动。
6. 需要时暂停或打开历史面板。

### 6.3 演示流程

1. 播放一段英文技术演讲。
2. 打开听译窗，展示源语言和目标语言选择。
3. 保持默认“自动识别 -> 简体中文”，点击开始听译。
4. 底部出现悬浮中文字幕。
5. 展示字幕边听边出。
6. 选择一段上下文相关内容，展示前一句被自动修正。
7. 打开历史面板，展示修正后的完整字幕记录。
8. 结束时总结：产品不是在做一次性翻译，而是在帮助用户持续跟上外语内容。

## 7. 功能设计

### 7.1 主窗口

主窗口承担控制台角色，保持克制，不做复杂首页。

核心元素：

- 产品名称：听译窗
- 听译状态：未开始、聆听中、暂停、连接失败
- 主按钮：开始听译、暂停、继续
- 音频来源：麦克风
- 源语言：自动识别、英语、日语、韩语
- 目标语言：简体中文、英语、日语、韩语
- 历史入口
- 简单设置入口

主窗口不应像营销落地页，而应像一个随开随用的桌面工具。

### 7.2 悬浮字幕窗

悬浮字幕窗是产品的第一体验重点。

布局建议：

- 半透明深色背景
- 目标语言字幕为主体，默认显示中文字幕
- 底部或侧边显示很轻的状态点
- 无大面积装饰
- 支持拖动
- 支持置顶

字幕状态：

- 临时字幕：显示正在生成的译文
- 已确认字幕：一句话结束后的稳定译文
- 已修正字幕：系统用更准确版本替换前文

### 7.3 字幕历史面板

历史面板用于增强安全感，让用户知道错过的信息可以找回。

内容结构：

- 字幕段落标记
- 目标语言字幕
- 可选原文
- 修正状态

第一版只保留本次会话内历史，不做持久化。

### 7.4 修正能力

修正能力是题目要求中的关键点，也是产品差异点。

系统应维护一个短上下文缓冲区：

- 正在识别的音频片段
- 最近几句原文
- 最近几句目标语言译文
- 字幕句子的稳定状态

当新音频到来后，系统可以重新生成最近一到三句翻译，并判断是否替换旧字幕。

字幕状态模型：

```text
draft -> stable -> revised
```

含义：

- `draft`：实时生成中，可能变化。
- `stable`：一句话结束，已判断为稳定。
- `revised`：后续上下文触发了修正。

为避免字幕频繁跳动，应限制修正范围：

- 只修正最近一到三句。
- 超过修正窗口的字幕不再修改。
- 修正幅度较小时不必更新 UI。

## 9. 交互细节

### 9.1 开始听译

用户点击开始后：

- 按钮变为暂停。
- 状态变为“聆听中”。
- 悬浮字幕窗出现或激活。
- 若没有权限，提示用户授权麦克风。

### 9.2 暂停听译

用户点击暂停后：

- 停止采集音频。
- 保留已显示字幕。
- 历史不清空。
- 用户可以继续或结束。

### 9.3 字幕更新

字幕更新分两种：

- 新增：出现新的字幕句子。
- 修正：替换最近一句或几句字幕。

修正时建议使用轻微过渡效果，例如短暂高亮，但不要用强提示打断用户观看。

### 9.4 错误状态

需要处理的错误：

- 麦克风权限被拒绝。
- 网络或 AI 服务连接失败。
- 持续没有检测到音频。
- API 返回异常。

错误提示应短而明确，例如：

- 未检测到音频
- 麦克风权限未开启
- 连接失败，点击重试

## 10. 技术方案概览

### 10.1 技术栈

建议采用 Electron、React、TypeScript 实现桌面悬浮窗口和本地控制界面。

### 10.2 模块划分

建议模块：

- 音频采集模块：负责麦克风或系统音频输入。
- 实时识别模块：根据源语言设置将音频流转为文本。
- 实时翻译模块：根据目标语言设置翻译识别文本。
- 修正引擎：根据上下文更新最近字幕。
- 字幕状态管理：维护 draft、stable、revised 状态。
- UI 展示模块：主窗口、悬浮字幕窗、历史面板。

### 10.3 数据结构草案

```ts
type SubtitleStatus = 'draft' | 'stable' | 'revised'

interface SubtitleItem {
  id: string
  sourceLanguage: string
  targetLanguage: string
  sourceText?: string
  translatedText: string
  status: SubtitleStatus
  startedAt: number
  endedAt?: number
  revisionOf?: string
}
```

### 10.4 修正逻辑草案

```text
1. 接收新的语音识别增量。
2. 更新 draft 字幕。
3. 当检测到语义停顿或句子结束时，将 draft 转为 stable。
4. 将最近几句 stable 字幕和新上下文送入翻译修正流程。
5. 如果返回的新译文明显更准确，则更新对应字幕为 revised。
6. UI 替换旧字幕，并在历史面板保留最终版本。
```

## 11. 视觉与体验方向

### 11.1 关键词

- 轻
- 清楚
- 稳定
- 低打扰
- 专注内容

### 11.2 视觉风格

建议采用桌面工具风格：

- 深色半透明字幕窗
- 高对比白色目标语言字幕，默认中文
- 简洁控制按钮
- 少装饰
- 少大面积营销式排版

### 11.3 字幕窗建议

默认放在屏幕底部中央，宽度适中。

字幕字体需要足够清晰，避免过小。窗口高度不宜太高，最多显示两行主字幕和一行上一句。

## 12. 成功指标

MVP 阶段不追求商业指标，重点验证体验闭环。

### 12.1 Demo 成功标准

- 能从音频输入中实时产生目标语言字幕，默认中文。
- 能在主窗口选择源语言和目标语言。
- 字幕延迟可接受，用户能感到跟得上。
- 能展示至少一次前文字幕自动修正。
- 悬浮窗体验完整，可置顶或模拟置顶。
- 历史面板能查看最近字幕。

### 12.2 用户体验标准

- 用户不需要学习复杂操作。
- 主流程不超过一次点击即可开始。
- 字幕不会频繁大幅跳动。
- 出错时用户知道发生了什么。

## 13. 实施阶段

### 阶段一：跑通核心链路

目标：让音频输入到目标语言字幕输出的链路能跑起来。

任务：

- 搭建主界面。
- 完成源语言和目标语言选择控件。
- 完成麦克风权限与音频采集。
- 接入语音识别与翻译能力。
- 完成基础字幕列表状态。
- 实现开始、暂停、继续。

验收：

- 用户点击开始后，能看到目标语言翻译不断出现。

### 阶段二：做出产品形态

目标：把工具做成“可以用”的桌面字幕助手。

任务：

- 实现悬浮字幕窗。
- 实现历史面板。
- 实现字幕状态：draft、stable。
- 优化实时字幕刷新节奏。
- 补齐错误状态。

验收：

- 用户可以边播放外语内容边看悬浮目标语言字幕。

### 阶段三：打磨修正能力与演示

目标：让产品亮点成立，并准备比赛展示。

任务：

- 实现最近字幕修正能力。
- 加入 revised 状态和 UI 高亮。
- 准备演示音频或视频素材。
- 打磨主界面与字幕窗视觉。
- 完成 README 与演示说明。

验收：

- Demo 中可以清晰展示实时翻译和自动修正。

## 14. 风险与取舍

### 14.1 系统音频采集风险

不同系统对系统音频采集支持不同，实现成本较高。

取舍：

MVP 优先麦克风输入。演示时可通过电脑外放加麦克风采集完成闭环，之后再扩展系统音频。

### 14.2 实时质量风险

实时识别和翻译可能出现延迟或错误。

取舍：

产品机制上接受 draft 字幕不完美，用 stable 和 revised 状态提升最终理解质量。

### 14.3 修正体验风险

频繁修正会让字幕跳动，影响观看。

取舍：

只修正最近一到三句，并控制修正频率。小幅改动不触发明显 UI 更新。

## 15. 未来方向

MVP 成功后可以扩展：

- 支持系统音频直采。
- 支持更多语言和地区变体。
- 支持双语对照。
- 支持术语表。
- 支持字幕导出。
- 支持摘要和关键点回顾。
- 支持本地模型或隐私模式。

## 16. 产品总结

听译窗的核心不是“翻译一句话”，而是“持续帮用户跟上一个外语内容流”。

MVP 应该呈现为一个极简、可演示、可依赖的桌面字幕助手：打开、开始、看字幕、自动修正、回看历史。

只要这个闭环顺滑，产品就已经成立。
